# 發現報告:調度員「接續下一位乘客」的實證規則(2026-07-04)

**資料來源**:`dispatch_history`(人工派遣歷史)——47,997 筆有座標+時間+車牌的成行紀錄,
316 個服務日,範圍 2024-11-01 ～ 2026-07-02,車行以台北(27,098)/新北(16,520)/神同行(4,172)為主。

**方法**:同一 `(車牌, 服務日)` 依 `pickup_time` 排序,取連續兩趟 A→B 統計接續特徵;
往返以「地理反向」偵測(座標四捨五入 ~110m,上車≈前趟下車、下車≈前趟上車、時間在後)。

**可重跑腳本**(純唯讀,不寫任何表):
```bash
docker compose exec -T backend python -m scripts.analyze_chaining    # 全局六規則 + 間距分佈
docker compose exec -T backend python -m scripts.analyze_chaining2   # ①分車行 ②來回趟
```

## 一、六條實證規則(全局)

| # | 規則 | 實測值 |
|---|---|---|
| 1 | **就近串趟**:下一位在剛下車點附近 | A下車→B上車空車距 **中位 2.7 km**(Q1 1.2、Q3 5.0、P90 8.1) |
| 2 | **約 40–80 分接下一位** | 上車間距 **中位 55 分**(Q1 40、Q3 80);眾數 30–45 分(27%) |
| 3 | **幾乎不共乘** | 與前趟時間重疊僅 **1.9%**;省車靠串趟不靠併車 |
| 4 | **同區是傾向非硬規則** | A下車與B上車同行政區僅 **39%**;每車日主導區僅佔 37.5% 趟 |
| 5 | **一台車一天約 8 趟** | 每車每日趟數 **中位 8**(Q1 6、Q3 9、P90 11) |
| 6 | **福祉車彈性兼一般單** | 有福祉單的車日中僅 **5.5%「整天只做福祉」**(94.5% 混做) |

**上車間距分佈**:0–15 分 3.9%、15–30 分 5.1%、**30–45 分 27.2%**、45–60 分 14.3%、
60–90 分 25.7%、90–120 分 11.0%、120 分以上 12.8%。

## 二、分車行對照——都會 vs 遠郊是兩種模式

| 車行 | 趟數 | 上車間距中位 | 空車距中位 | 同區% | 共乘% | 每車日趟 |
|---|--:|--:|--:|--:|--:|--:|
| 台北 | 27,098 | 60 分 | 2.5 km | 36% | 1.1% | 5 趟 |
| 新北 | 16,520 | 60 分 | 3.5 km | 37% | 1.6% | 4 趟 |
| **神同行** | 4,172 | **30 分** | **0.8 km** | **98%** | **8.0%** | **12 趟** |
| 其他(校車/日照…) | 207 | 480 分 | 0 km | 78% | 1.2% | 2 趟 |

- **都會**(台北/新北)= 疏散點對點:60 分節奏、跨 2.5–3.5 km、幾乎不共乘、4–5 趟/車日。
- **神同行**(金山/萬里遠郊)= 密集在地接駁:30 分接一位、空車 0.8 km、**98% 同區**、
  共乘率 8%(全隊最高)、12 趟/車日。**必須分車行校準**,全域參數會嚴重誤判。
- 「其他」= 校車/日照等**既定區塊固定趟**(480 分間距、2 趟/日),本就稀疏。

> 印證:原則8(分車行×福祉校準)、原則9(既定區塊)、`_FLEET_REGION` 區域隔離。

## 三、來回趟——長照主結構是往返,不是單程

- **約 61.8% 的趟次屬往返配對**(30.9% 可辨識為「去程」且有對應回程)。
- **目的地停留中位 143 分**(Q1 103、Q3 269);**35% 停留 >180 分**(洗腎 ~4h/復健/回診)。
  停留分佈:<30 分 0.4%、30–60 分 6.1%、60–90 分 13.3%、90–120 分 16.6%、
  120–180 分 28.5%、180 分以上 35.1%。
- **回程由「同一台車」承接僅 41.8%** → 近六成回程換了別台車。
- 分車行往返率:神同行 **49.4%** ≫ 台北 29.8%、新北 28.1%。

**營運解讀**:調度員**不讓車空等 2.4 小時**——送病人到院後,車立刻去接別的單,
2 小時後由當下最順路的車(多半是別台)回接。**省車的核心是用「停留空檔」填別人的趟,而非併車。**

**系統含意**:VROOM 把去程/回程當兩張獨立 shipment(各自時間窗),回程自然指派給
該時點最省的車 → **現行引擎已天然重現「58% 回程換車」**,方向正確。

## 四、對系統的總結含意

1. 現行引擎(VROOM 最小化總車程 + 原則8 分車行校準 + 原則9 既定區塊)**已吻合這六條的方向**。
2. 可再優化的唯一旋鈕:**區域親和「分車行差異化」**——對神同行加重(98% 同區)、
   對都會維持弱(36–37% 同區);而非全域統一權重(與 BACKLOG「區域親和權重要保守」一致,
   但資料顯示應做成**分車行差異化**)。建議先做**唯讀 what-if** 量測影響再決定是否開。
3. 共乘同意的低影響已再獲印證(都會併車 ~1%),保守預設「不同意」符合實況。

## 五、人工 vs 系統自動 實測比對(同 271 日、同指標)

來源:人工 `dispatch_history` vs 自動 `auto_dispatch_stop`(已落地),限共同服務日。
重跑:`docker compose exec -T backend python -m scripts.compare_human_auto`

| 指標 | 人工 | 自動 | 解讀 |
|---|--:|--:|---|
| 每車每日趟數(中位) | 8 | 7 | 幾乎一致 |
| 連續上車間距(中位) | 55 分 | 60 分 | 幾乎一致(自動略鬆=保守設定) |
| 連續上車點距離(中位) | 3.4 km | 5.0 km | ⚠️ 自動接續點較遠(全域最佳化 vs 就近) |
| **跨單併車率** | **1.7%** | **0.0%** | **兩者皆點對點,一致**(神同行除外) |
| 往返回程同車% | 41.6% | 50.8% | 自動較常同車回接 |
| 總車日(全公司去重) | 5,180 | 5,045 | 自動省 ~2.6% |

> ⚠️ **併車率指標更正(2026-07-04)**:初版曾誤報「自動併車 25.7% vs 人工 3.3%」,
> 係把「單一訂單 2 人(病患+陪同)」誤計為併車(occupancy 計人數)。改用**跨單指標**
> (在車有『其他訂單』;`scripts/pool_source.py`)後,**人工 1.7% / 自動 0.0%——兩者皆點對點,一致**。
> 驗證:單車路線序為「上車→下車歸零→再上車」,occ=2 來自該單本身 2 人,非兩單同車。

**吻合**(系統已複製人工):時間節奏、趟數密度、**併車(皆點對點)**、往返結構(送去→接回、回程多換車)。

**分歧**:
1. **接續點 3.4→5.0km**:人工貪婪就近、系統全域最佳化(單車路線看似繞較遠,整體更省)。
2. **神同行(遠郊)併車**:人工併車 14–17%、自動 0%——那些單無共乘同意 → 被強制獨佔,
   造成當地多幾筆未派(印證「遠郊金山/萬里靠共乘」)。**真正該處理的是這裡:對神同行放寬併車**,
   而非全域收緊。(what-if:`force_no_pool` 對現況無作用,因非固定趟本已獨佔;`scripts/whatif_no_pool.py`)
   - **開放共乘 what-if(30 抽樣日,`scripts/whatif_fleet_pool.py 神同行`)**:
     現況用車 60/未派 127/併 0% → 開放共乘 用車 59/**未派 80**/併 23.5%。
     **未派 −47 趟(−37%)、用車 −1 台**。剩餘未派多為服務時段外/座標疑誤/遠程專車(非共乘可救)。
     → 落地方式:對神同行訂單設 `pool_consent`(或加「車行層級自動同意」設定),需政策/合規拍板。
3. **省車兩種算法**:全公司去重 5,180→5,045(2.6%)vs 逐車行 7,928→7,408(6.6%);
   差在「一車服務多車行」去重只算一次。對外報表用逐車行值,實際少開幾台用去重值。
