# 評估:routingpy / Valhalla(時段相依路由)是否值得導入

> 結論先講:**DEFER(暫不導入)**。我們資料證實 08:00 / 17:00 有真實通勤壅塞(同距離下慢約 40%),
> 但 ① VROOM 吃**靜態車程矩陣**,架構上無法消費「時段相依」車程;② 現有營運緩衝已吸收該誤差;
> ③ Valhalla 需自架第二套路由引擎 + 台灣圖磚,成本高。routingpy(多供應商抽象)目前無需求。
> 評估日期:2026-06。資料:42,907 趟真實派遣紀錄(含 distance_m / est_minutes)。

## 動機
GitHub 探勘時兩個路由相關候選:
- **Valhalla**(`nilsnolde/docker-valhalla` 317★ / `pyvalhalla` 70★):賣點是**時段相依路由**(尖峰/離峰不同車程)。
- **routingpy**(342★):一個 Python 庫統一接 OSRM/ORS/Valhalla/Google,等於幫路由做「provider 可切換」。

要先用資料證明「尖峰真的塞、且值得處理」,再決定。

## 資料發現一:確有通勤尖峰壅塞(但不在我們的量峰)
各時段隱含中位時速(距離 ÷ 估時,n=42,417):

| 時段 | 中位時速 | vs 全日 |
|---|---|---|
| 07 | 23.2 | −4% |
| **08** | **14.8** | **−39%** ← 早通勤尖峰 |
| 09–13(就醫量峰) | 22–27 | 正常 |
| 16 | 23.1 | −5% |
| **17** | **13.6** | **−44%** ← 晚通勤尖峰 |

**關鍵**:我們的**訂單量峰是 09–13(就醫/復健),車速正常**;真正塞的是 **08:00 與 17:00 通勤尖峰**。

## 資料發現二:控制距離後確認是真壅塞(非短程市區拉低均速)
同一距離帶下的中位時速:

| 距離帶 | 07時 | 08時 | 11時 | 13時 | 17時 |
|---|---|---|---|---|---|
| 2–5 km | 23.2 | **14.3** | 23.2 | 24.7 | **14.1** |
| 5–10 km | 31.8 | **20.0** | 31.6 | 32.6 | **19.1** |

→ 同距離下 08/17 仍比 11/13 慢約 38–40%,排除距離混入,**確為時段壅塞**。
影響範圍:08 時占 6.1%、17 時占 7.0% ≈ **全部趟次的 ~13%**。

## 為什麼仍 DEFER

### ① 架構擋路:VROOM 吃靜態矩陣
`dispatcher.py` → `matrix_svc.build_matrix()`(OSRM `/table` 一次算好)→
`problem.set_durations_matrix("car", arr)`。**VROOM 用單一、時段無關的車程矩陣**;
同一段路無法依「車輛幾點經過」給不同車程。
即使接了 Valhalla 拿到時段相依車程,**現有求解器也消費不了** —— 要真用,得換時段相依求解器或大改,成本遠超效益。

### ② 營運緩衝已吸收該誤差
現行:上車時間窗 **±30 分** + 每趟前後置作業 **40 分**。
典型趟程 6–16 分;尖峰低估 40% 對一趟 15 分的單僅 +6 分,**遠小於緩衝**。
這也是為何用 OSRM 自由流仍能跑出 ↓18.6% 且未派極少。

### ③ 基礎設施成本高
Valhalla = 自架**第二套路由引擎** + 下載/維護**台灣圖磚**(同 OSRM 等級的 GB 級資料 + 容器),
為了 ~13% 趟次、且緩衝已吸收的誤差,不符「精簡」原則。

### routingpy
是「多供應商可切換」的抽象層;我們只用 OSRM、無切換需求,**現在導入是為不存在的問題加複雜度**。

## 建議
1. **Valhalla / 時段相依路由:DEFER**。把上述壅塞發現留作營運情報(例:08:00/17:00 避免過度緊湊串接、或未來餵進緩衝邏輯)。
2. **routingpy:DEFER**。真要換路由供應商時再評估。
3. **重新評估的觸發條件**:① 改用時段相依求解器;② 收緊緩衝 / 上準點 SLA;③ 客訴集中在 08/17 遲到;④ 擴張到壅塞更嚴重的區域。
