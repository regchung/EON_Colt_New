## 三、資料告訴我們的事(實測洞察)

- **成行率**:已轉正式單 67.5% / 已重置 32.3% → 決策分析只取「已轉正式單」。
- **常客固定駕駛(有訊號✅)**:部分乘客高度集中同一駕駛(68% / 39% / 29%)→ 值得做成排班規則。
- **駕駛熟區(偏弱⚠️)**:最常跑的區也只佔 11–21% → 司機雙北全域跑,**區域親和權重要先量測再決定**。

## 四、⚠️ 目前的資料治理問題(要處理)

1. **歷史單污染營運表**:2 萬筆歷史單進了 `orders`。多數 status=done/canceled(排班會排除),但有 28 筆「已有距離及時間」落到 `imported`,且**報表/列表會把 2 萬筆全算進去**。
   → 建議:`orders` 加 `is_historical`(或 `source`)欄位區分,營運視圖預設只看非歷史。
2. **車隊主檔混四個車隊**:47 車/41 司機來自四個不同車隊,全 active → 一鍵排班會混排。
   → 建議:加「車隊/子車隊」歸屬,排班時挑單一車隊;或多租戶隔離。
3. **截斷資料**:新北/台北各缺 1 萬列以後 → 分段(月/季)重新匯出疊加(冪等)。
4. **地址簿文字瑕疵**:少數來源地址重複區名(如「內湖區內湖區…」)→ 同址被當不同門牌(座標仍正確)。可跑 Map8 standardization 合併別名。
5. **垃圾車輛**:一筆佔位車牌「X」建成車輛 → 應刪。

## 五、後續優化建議(分類)

### A. 資料治理
- `orders` 與 `vehicles/drivers` 加 `source` / `is_historical` / `fleet`,**歷史與營運分流**。
- 匯入加 `import_batch`(來源檔、時間),可追溯、可回滾某次匯入。
- 個資:電話遮罩政策、保存期限、去識別化送外部(Claude)時遮罩。

### B. 經驗引擎(把人工變規則)
- **常客固定駕駛** → `/dispatch/driver-suggest`(類比 zone-suggest,資料支持度最高,優先做)。
- **駕駛熟區 / 派單員習慣 / 時段 / 共乘** → 萃取成 `dispatch_rule` 表;熟區先量測。
- 規則餵進建議與 VROOM 軟性偏好。

### C. 排班引擎
- 區域親和接進 VROOM **軟性偏好** + **人工 vs 自動對比**(車數/里程/準點)。
- **時段路況矩陣**:用歷史「距離/預估分鐘」校準 OSRM(尖峰加成)。
- 重排時 `ongoing` 用 VROOM `steps` 強制鎖定(目前是排除簡化版)。

### D. 效能 / 工程
- 大量匯入優化:目前 1 萬列同步 ~80s。改 **批次 insert + 記憶體快取地址/車/司機 + 背景任務(SSE 進度)**。
- `dispatch_history.raw` 改 **JSONB + GIN 索引**(目前 JSON,查 raw 欄位較慢)。
- zone_affinity / assign / history 補 pytest,CI 守穩。

### E. 產品 / 前端
- **歷史匯入 UI**(上傳 + 報告 + **涵蓋報表**:子車隊×月份,看缺口)。
- **經驗洞察頁**:駕駛熱力圖、常客固定駕駛、派單員分布。
- **人工 vs 自動對比頁**(對外賣點)。

### F. 商業化 / 部署
- **多租戶隔離**(各車隊資料分離)——賣多家車隊的硬需求,現在四隊混存。
- 正式部署:改 `SECRET_KEY`/密碼、HTTPS、各環境 `.env`、備份。

## 六、建議優先序

1. **資料分流**(orders/vehicles 加 source 旗標)+ 刪垃圾車「X」 — 讓營運視圖乾淨。
2. **常客固定駕駛建議**(driver-suggest)— 資料支持度最高、最快有感。
3. **人工 vs 自動對比** — 證明打敗人工(賣點)。
4. **歷史匯入 UI + 涵蓋報表** — 讓車隊自助分段補匯。
5. **多租戶 + 部署** — 商業化前硬需求。
