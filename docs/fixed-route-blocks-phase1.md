# 固定行程「既定骨架 + 衝突偵測」(階段一)— 實作與驗證結果

## 背景與設計
提案:固定行程由指定司機執行 = **既定承諾**,不該丟進 VROOM 一起最佳化(靠脆弱的 skill-pin),
而應當成各司機當天的**既定時間骨架**;系統只需:
1. 把每位指定司機的固定趟次排成骨架,算每趟**佔用窗**(上車前置 → 行程 → 下車後置);
2. **偵測衝突**:同司機兩趟時間重疊(單車無法分身)或銜接不及;
3. 算每位司機**可接單空檔**(供階段二把彈性單塞進空檔)。

階段一**純唯讀分析,不改現行派遣**(零風險):
- `services/fixed_route_blocks.py`:`analyze(db, service_date)`。
- `GET /fixed-routes/blocks?service_date=`(需 dispatcher 以上)。
- 車程用 OSRM;CI 無 OSRM 時退回 haversine,確保不中斷。

## 驗證結果(2026-06-22 真實資料)
| 指標 | 數值 |
|---|---|
| 固定趟次 | 43 |
| 指定司機(車) | 17 |
| **偵測到衝突** | **9 件,集中在 4 位司機** |
| 固定司機可接單空檔 | **約 176 車·小時** |

### 衝突明細(自動分類處理方向)
| 車 | 衝突 | 分類 |
|---|---|---|
| 197(成德國中) | 07:00×2、16:00×2 重疊 | 🔁 同校同時段 → **可共乘併車** |
| 190(百齡高中) | 07:10×2、15:50×2 重疊 | 🔁 可共乘 / ⚠️ 需備援 |
| 209(潭美國小) | 07:00×2、15:50/16:00 重疊 | 🔁 可共乘 |
| 228(向怡診所) | 10:30/11:20/11:40、16:00 重疊(**4 人團**) | ⚠️ 滿座無法併 → **必須加備援車** |

### 與舊行為對照(關鍵價值)
- **舊**(VROOM + skill-pin):這些超載趟次被默默丟進「未派」(6/22 的 11 筆),行控不易看出原因。
- **新**(骨架 + 衝突偵測):同一批問題變成 **9 件明確衝突**,且自動標示:
  - 學校群(成德/潭美/百齡)=「**同校同時段、座位夠 → 可共乘**」;
  - 向怡診所(4 人團)=「**滿座無法併 → 需加備援車**」。
  → 行控一眼看到該怎麼處理,而非埋在最佳化結果裡。

## 結論
階段一達成兩個確定價值:**① 把超載從「靜默丟單」變成「明確衝突 + 處理方向」;② 量化固定司機閒置(176 車·時)**。
全程不影響現行派遣;可作為行控的「固定行程健檢」。
