# SmartCar 自建功能規劃(調度智慧化)

> 目標:解決「自動排班輸人工」的核心痛點(缺**地理 / 經驗知識** + 缺**人在迴路**),
> 但**不外採重型服務**(Dify/RAGflow),改用現有元件(FastAPI + Vue + Claude + VROOM + OSRM + Map8)自建。

## 設計原則(為何自建)

1. **產品要賣給車隊 → 部署要輕**:每多一個重服務(Dify ~10 容器、RAGflow 需 16–32GB)就多一份維運與成本,也踩授權雷(Dify 多租戶 SaaS 受限)。自建讓部署維持 4 服務。
2. **壓榨現有元件**:Claude(`ai_dispatch.py`)、VROOM、OSRM、地址簿都已就緒,先補缺口再加服務。
3. **人在迴路(augment, not replace)**:AI 出建議、調度員確認後才寫入,信任感與可控性是賣點。
4. **資料驅動、漸進**:四月資料量小,先用「摘要 + prompt + 規則約束」拿 80% 效果,資料累積後再上 RAG。
5. **可解釋**:每個自動決策都要能對調度員「說清楚為什麼」,這是贏過黑箱的關鍵。

外採決策(見 `doc` 內整合評估):**MarkItDown=函式庫直接用**;**Dify/RAGflow=概念自建、暫不導入**。

#### 首個具體規則:區域親和(Zone Affinity)— 屬 C1/C2,優先實作

**規則(原始描述)**:「若一個司機今天已有 N 筆在某區域的單,新進來的同區域單優先給他,不管時間順序。」

**修正為可安全落地的版本(軟約束 + 護欄)**:
> 在**不違反時間窗、車種/輪椅、座位**的前提下,讓**同區域的新單優先指派給今天已在該區累積 ≥ N 筆的司機**;偏好為「**加權**」而非「硬性」,並設每人每區上限。
> 「不管時間順序」解讀為「**區域親和的權重高於純距離/時間排序**」,而非忽略可行性。

**為何有用**:減少空車里程、發揮熟區效益(巷弄/出入口/時段路況)、常客連續性、貼近人工直覺且可解釋——正是 VROOM 預設不會做的「跨整天把同一區黏同一司機」。

**護欄(硬約束,絕不破壞)**:
1. 時間窗可行性(長照準時為命脈)。
2. 車種 / 輪椅 skill。
3. 座位 capacity。
4. 每人每區上限 `max_jobs_per_zone`,避免過載失衡。

**區域定義**:優先用 `address_point.city/town`(Map8 行政區,已有),之後可細化為地理網格/分群。

**資料來源(已備齊)**:`orders.assigned_vehicle_id` + 該單地址 town 即可算「司機今天在某區幾筆」;司機已綁 `users.vehicle_id`。

**落地點(由易到難)**:
- **(優先)動態插單**:新單進來時,`/dispatch/ai-insert` 先找「已在該區且可行」的司機,可行就優先給;不可行才回退最佳化。風險最低、最貼近情境。
- **批次排班**:當成 VROOM 軟性成本偏好(同區同車折扣 / zone-first 兩階段),須搭 D2 人工 vs 自動對比量測後再開大。

**可調參數(交給車隊調)**:
- `zone_affinity_weight`:親和偏好強度(0=關閉,越大越黏區)。
- `min_jobs_N`:達 N 筆才觸發偏好(預設 2)。
- `max_jobs_per_zone`:每人每區上限(避免過載)。

**驗收**:同區新單在可行時優先落到既有司機;時間窗/車種/座位永不被破壞;可一鍵關閉回退純 VROOM。

## 自建 vs 外採對照(決策依據)

| 能力 | 外採選項 | 自建做法 | 採用 |
|------|----------|----------|------|
| 文件轉文字 | Azure Doc Intelligence | **MarkItDown(lib)** | 自建(用 lib) |
| 對話 agent 編排 | Dify | **Claude tool-use + 自家 UI** | 自建 |
| 歷史知識問答 | RAGflow | **規則萃取 + prompt;之後再 RAG** | 自建(暫) |
| 路網/矩陣 | 付費 API | **自架 OSRM** | 已自建 |
| 地理編碼 | — | **Map8 + 地址簿快取** | 已自建 |

## 階段時程(建議)

**Phase 1(快贏,~1 週)** — P0 為主
- A1 文件智慧匯入、E2 導航深連結、F4 部署強化
- D1 排班解釋(小而有感)

**Phase 2(核心價值,~2–3 週)** — 調度智慧化
- B1–B4 調度員 AI 助理(對話 + 工具呼叫 + 確認寫入 + 插單)
- C1–C2 歷史規則 → VROOM 約束
- E1 狀態回報→重排、D3 KPI 看板

**Phase 3(進階 / 商業化,~3–4 週)**
- C3 時段路況、A2/A3 匯入校正與記憶
- F1 多租戶、F3 個資合規、D2 人工 vs 自動對比
- E3 推播

**Phase 4(視資料量)**
- C5 非結構化 RAG(累積文字資料後再評估 RAGflow 混合)

## 與現有程式的銜接點(實作時對照)
- 匯入:`backend/app/services/importer.py`、`api/routes/orders.py`
- 排班:`backend/app/services/dispatcher.py`(組 VROOM 問題處加經驗約束)
- AI:`backend/app/services/ai_dispatch.py`(擴充為助理 + 工具呼叫)
- 矩陣:`backend/app/services/{osrm,matrix}.py`(時段加成)
- 地址:`backend/app/services/geocode.py`、`map8.py`(已含 standardize)
- 前端:`frontend/src/views/`(新增助理面板、匯入預覽、KPI、司機導航)

> 原則重申:**先用現有元件補缺口,能不加服務就不加**;Dify/RAGflow 僅在「自建成本 > 外採成本」且授權/部署可接受時才導入。
