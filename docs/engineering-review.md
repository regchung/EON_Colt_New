# DrFish 軟體工程檢視

> 以軟體工程角度對本案的誠實評估(優點與風險並陳,附嚴重度與建議)。
> 檢視基準:migration 0023、18 路由 / 26 服務 / 15 資料表 / 20 前端頁、56 tests、CI 綠。

## 整體判斷
**工程紀律罕見地好的 MVP/試點級系統,但尚未達「商業 SaaS 上線」標準。**
缺口恰好是 backlog 標的「收費前硬需求」+ 核心測試覆蓋 + 可觀測性。

---

## ✅ 做得好的(超出同級專案)
1. **證據驅動的技術決策** —— PyVRP(NO-GO)、類神經網路(實測更差)、Valhalla(DEFER)
   皆有資料與評估文件背書。多數團隊靠感覺,本案靠數字。**最大亮點。**
2. **分層乾淨** —— routes(薄)→ services(邏輯)→ models/schemas;分析型服務唯讀、與寫入路徑分離。
3. **務實選型** —— VROOM 原生吻合 PDPTW(且驗證過 PyVRP 不支援才沒換)。
4. **優雅降級無所不在** —— 無金鑰 AI、無 OSRM 退 haversine、失效推播自動清。
5. **依賴衛生** —— 16/17 釘死版本;CI 三關綠;0 個 TODO/FIXME 殘留;無 SQL 注入;機密進 `.env`。
6. **時區全鏈一致 +08** —— 這類 bug 已獵到根因並收尾。

## 🔴 上線前必須處理(P0)
| # | 問題 | 說明 / 建議 |
|---|---|---|
| 1 | **多租戶隔離不存在** | 全部車行共用一個 DB,`fleet` 只是標籤、**非安全邊界**;查詢不依租戶過濾。賣多家車隊的第一風險。建議 `tenant_id` + 查詢層強制過濾,或 Postgres RLS。 |
| 2 | **個資外送第三方** | AI 文件匯入/助理把乘客姓名·地址·電話原文送 Claude。個資法合規風險,收費前需地端抽取或簽 DPA。 |
| 3 | **預設機密** | `SECRET_KEY="change-me..."`、`admin123` 靠 env 覆寫;漏一個就被攻破。建議啟動時偵測預設值即拒絕啟動。 |

## 🟠 顯著風險(P1)
| # | 問題 | 說明 / 建議 |
|---|---|---|
| 4 | **核心路徑零測試** | 4 測試檔 / 26 服務;最關鍵複雜的 `dispatcher.py`(342 行)求解、`comparison`、`pool_suggest` **完全沒測**(CI 無 OSRM)。56 測試測的是容易的部分。建議 CI 備小型 OSRM 或 mock 矩陣,補求解整合測試。 |
| 5 | **無速率限制** | 登入(暴力破解)、AI 端點(成本濫用)裸奔。加 `slowapi`。 |
| 6 | **可觀測性空白** | 無結構化日誌/metrics/錯誤追蹤(Sentry/OTel)。上線出事將無從診斷。 |
| 7 | **擴展天花板** | VROOM 靜態矩陣 + 同步 OSRM + 單一 DB。試點夠,規模化需重構(routing 評估已述)。 |
| 8 | **Map8 金鑰對前端外露** | `/config` 回傳給客戶端(地圖固有需求),但該金鑰應做網域/referrer 限制。 |

## 🟡 衛生問題(P2)
- **22 處 broad except** —— 多為刻意降級,但部分靜默吞錯(matrix fallback、推播)應補日誌。
- **前端 god component** —— `Orders.vue` 715 行;單一 bundle 1.27MB 未 code-split。
- **alembic 漂移**(已修)—— 建議補「斷言所有 model 模組被匯入」的測試防回歸。
- **106 同步端點 / 5 async** —— I/O-bound 低併發可接受(threadpool),高負載限吞吐。

## 架構評語
- 「**固定行程當骨架、與最佳化解耦**」方向正確 —— 確定性承諾與最佳化分離,是成熟設計。
- service 層因子化良好、易測(但核心未測)。

## 優先建議(收費前)
1. **多租戶隔離 + 個資地端化 + 強制機密**(backlog 已列,門檻項)。
2. **補 dispatcher 整合測試**(核心零覆蓋是最危險的技術債)。
3. 加**速率限制 + 結構化日誌 + 錯誤追蹤**。
4. 前端 code-split、拆 `Orders.vue`。

---
**一句話**:技術判斷力與工程紀律是本案護城河之一;但「能 demo」到「能賣多租戶 SaaS」之間卡著
**隔離 / 合規 / 核心測試 / 可觀測性**四道,且都是已知項 —— 補上即為紮實產品。
