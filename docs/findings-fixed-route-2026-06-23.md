# 固定趟派遣處理報告 — 客戶回饋與既定區塊模型

**日期**:2026-06-23 ｜ **範圍**:EON COLT 派遣對比引擎 ｜ **資料**:雙北/基隆長照車隊歷史 47,184 筆 + 6/23 班表

---

## 摘要(一頁版)

客戶指出「固定趟(校車、日照、洗腎等已談妥行程)被系統當成一般散單亂併,虛報省車且忽略實況」。
經查證屬實:固定趟在班表中以「假單」進件、只記頭尾、時間為假設,原對比引擎會把多條「同到某校/某機構」的固定趟併成少數車。

我們導入**「既定區塊(committed block)」模型**:固定趟釘到指定車、以整趟佔用時間計、各趟各釘各車不互併,正常散單仍照常最佳化。

| 指標 | 結果 |
|---|---|
| 歷史散單對比省車率(556 天、768 組) | **↓34.9%**(人工 7,637 車·日 → 系統 4,971) |
| 校車(6/23,17 條) | 散單模式亂併成 **10 車(失真)** → 既定區塊 **16 車(正確)** |
| 日照(6/23,2 條) | 亂併成 **1 車(失真)** → 既定區塊 **2 車(正確)** |
| 既有 4.7 萬筆歷史 | 新模型**完全無影響**(向後相容) |

---

## 1. 問題(客戶回饋)

客戶於檢視 6/23 自動派遣後提出四點:

1. **固定趟是共乘單,彼此不可綁在一起。** 每一張固定趟訂單本身就是一張「一車多人」的共乘單(如「小驢駒先生A」內含數名乘客);因此「小驢駒先生A」與「小驢駒先生B」、「潭美國小-1」與「潭美國小-2」**不能被系統併到同一台車**。
2. **只輸入頭尾、中間省略 → AI 誤判。** 固定趟一次載很多人,過去只輸入起點與終點,中間站省略;系統因此以為整趟只有兩站、時間很短,**低估車輛佔用**,才會誤以為還有餘力去併更多單。客戶詢問:**訂單能否設定「佔用時間」或新增「中間站點」?**
3. **固定趟時間是假設值。** 校車/日照/洗腎等固定趟的時間目前只是假設,非實際時間;客戶詢問是否需要提供實際時間。
4. **固定趟盡量綁同一位駕駛。** 像校車這種已談妥的行程,通常固定由同一位駕駛服務,除非有狀況才換人。

### 問題根因(查證)

- 6/23 班表 244 列 = 正常 200 + 候補 7 + **假單 37**。那 **37 列假單正是固定趟**(子車隊=校車 33、日照 4),與系統內已建的 22 條固定行程一一對應。
- 匯入器(`schedule_import`)設計上**略過假單** → 固定趟根本沒進對比;而引擎對進件的訂單一律設 `allow_pool=True`(可併)。
- 資料模型為**單一起訖點**,**無「佔用時間」欄位、無中間站概念** → 印證客戶點 2 的低估問題。

---

## 2. 方案(評估的選項)

| 方案 | 內容 | 取捨 |
|---|---|---|
| **A. 既定區塊 + 釘司機(採用)** | 固定趟匯入為不可併、釘指定車/駕駛、以整趟佔用時間計的 committed block;正常單在其空檔最佳化 | 最貼近實況;需加 schema 欄位 + 引擎改動 |
| B. 僅標不可併、不釘車 | 固定趟也丟進同池但 `allow_pool=False` | 簡單,但不反映「佔住特定司機」 |
| C. 維持略過固定趟 | 固定趟不進最佳化,另行人工排 | 對比只涵蓋散單,無法評估固定趟對車隊的佔用 |

**距離矩陣**:採自建 OSRM(台灣真實路網,免費),非直線估算。
**工時**:採資料驅動的歷史校準(見第 4 節),非直覺固定值。
**決議**:採 **方案 A**;固定趟時間先以假設值跑通機制,待客戶提供實際時間再精修。

---

## 3. 解決的方式(實作)

### 3.1 資料模型
- `orders` 新增欄位 **`occupancy_min`**(固定趟整趟佔用分鐘)。
- 設值即視為「既定區塊」;`NULL` = 一般單,維持原行為。
- Migration **0027**;`alembic check` 通過、無 schema 漂移。

### 3.2 派遣/對比引擎(`compare_day`)
當訂單 `occupancy_min` 有值時:
- **釘車**:給每筆固定趟一個唯一 skill,只有其指定車具該 skill → 強制釘到 `assigned_vehicle_id`。
- **各釘各車**:不同固定趟用不同 skill → **彼此不會被併**(回應客戶點 1)。
- **佔用時間**:以 `occupancy_min` 當該趟服務時長(回應點 2 的「佔用時間」)。
- **解除最長乘車上限**:多站長趟不被誤判超時。
- **向後相容**:`occupancy_min IS NULL` 的 4.7 萬筆歷史訂單行為完全不變。

### 3.3 固定趟進件
- 6/23 的 37 條固定趟由「假單」改匯為既定區塊:**釘車 by 車牌**、建立 `已轉至正式單` 人工歷史作為對比基準。
- 佔用時間目前以「OSRM 頭尾車程 + 30 分多站(假設)」估算(校車均 44 分、日照均 40 分),**待客戶提供實際時間替換**。

### 3.4 駕駛綁定(點 4)
- 釘車採班表當日指定的車牌/駕駛 → 同趟綁同駕駛;當天若換人(如 6/23 小驢駒B由温智祥、文山特教由張啟明)亦如實反映。

## 5. 最終處理方式與結果

### 最終處理
1. **採用既定區塊模型**:固定趟釘車、計佔用時間、各釘各車不互併、綁定駕駛。
2. **散單照常最佳化**:一般長照叫車仍走 VROOM + OSRM 共乘最佳化(歷史已驗證省車 34.9%)。
3. **向後相容**:既有歷史與既有對比不受影響。
4. 程式碼已提交(`feat: 固定趟既定區塊模型`),並保留還原點 `restore-baseline`。

### 結果
- ✅ 客戶點 1(A、B 不能綁):固定趟各釘各車,確認不再互併。
- ✅ 客戶點 2(佔用時間/中間站):已新增「佔用時間」欄位並餵入引擎;中間站以佔用時間代理。
- ✅ 客戶點 4(綁同駕駛):以班表指定車牌/駕駛釘車,支援當天換人。
- 🟡 客戶點 3(實際時間):機制已就緒,目前用假設值;**待客戶提供固定趟實際整趟時長**即精準化。

### 後續建議
1. **收集固定趟實際時間**(首站上車時間 + 整趟佔用時長),替換假設值。
2. **跨車隊統一池**:讓校車/日照駕駛的午間空檔承接一般散單,進一步省車(目前對比仍各車隊獨立)。
3. **逐乘客建單(選配)**:若要精準到中間每一站,改以「一乘客一張單 + 固定行程釘車」呈現真實多站路線。
