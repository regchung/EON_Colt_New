# 派遣原則(Dispatch Principles)

即時派遣(`dispatcher.run_dispatch`)遵循的營運原則與其技術實現。
更新日期:2026-06。實現引擎:VROOM(PDPTW,shipment 模型)。

---

## 原則 1:先派固定趟、綁指定駕駛、且固定趟可共乘
- **只比對乘客姓名(不比對地址)**:固定規則以 `match_name`/`keyword` 比對**訂單 passenger_name**,
  **不再**比對地址/case_tag——避免「某地點所有乘客被灌給同一司機」造成過載/時間衝突而漏單。
  (案例:向怡診所地址規則曾把 6 趟全綁一台 4 座車、兩趟撞 16:00 → 必漏;改姓名 only 後當日 207 單全派、0 未派。)
- **綁指定駕駛(非綁死車號)**:固定行程於 `fixed_route.driver_name` 設「指定司機」;
  `fixed_route_match` 先用 `driver_resolve` 解出該司機**當天開哪台車**(當日輪車 > 司機預設車),
  再把固定趟釘到那台 → 指定司機換車,固定趟跟著走。
- **先派**:固定行程以**唯一技能硬綁**該車(`PIN_SKILL_BASE+vid`),`priority=80`(一般單 50)→ 優先排入,
  且只有該車能接(別車無此技能)。
- **邊界**:若指定司機當天查無車(`unresolved_rules`),該固定趟目前**退回一般派遣**(交其他可用車);
  若要改為「無指定司機則保留不派」,需另開選項。
- **可共乘**:固定趟訂單強制 `excl=1`(可與他人併車),**不受「共乘需同意」限制**——
  與一般單「未同意→獨佔整車(`excl=EXCL_CAP`)」不同。
- 程式:`dispatcher.py` excl 判定加 `if o.id in fixed_pins: excl = 1`。

## 原則 2:同一區域盡量放同一台車(軟性)
- 由 VROOM **最小化總車程**的目標函數自然達成:鄰近上/下車點會被聚到同一條路線/同一台車。
- **刻意做成軟性**(非硬牆):需要時仍可跨區調度,避免切斷高效共乘。
- 可調旋鈕:`zone_affinity`(區域親和,弱訊號)為未來要「加強」同區傾向時的權重來源,
  預設不硬接(見 BACKLOG「區域親和接進批次排班」)。

## 原則 3:訂單間 ≤ 1 小時車程、同一行政區優先(軟性)
- 同樣由「最小化總車程」目標達成:長腿(>1h)與跨行政區的接續會被自然懲罰、儘量避免。
- **刻意不設硬上限**:實測顯示收太緊會使未派上升、用車增加(與省車目標相反,
  見 `open-decisions.md` D1 單日 what-if)。保持軟性,讓引擎在效率與覆蓋間取最佳解。

## 原則 4:是否需福祉車「只看匯入時的車型文字」
- **判斷時機與依據**:**匯入時**看「車型」欄位的值——值帶有「福祉車」文字(含「福祉」)→
  此單**必須由福祉車接單**;其餘任何值(一般車/復康巴士/輪椅/空白…)→ **不限車種**。
- 落地:匯入 `_parse_vehicle_type` 把含「福祉/福祉車/welfare」的車型存為 `vehicle_type='welfare'`,
  其餘存 `'normal'`;`vehicle_type` **純由車型欄位決定**,不受「輪椅」欄位影響。
- 派遣:`dispatcher._is_welfare(o) = (o.vehicle_type == 'welfare')`,**不再**以 `need_wheelchair` 判定。
- 機制:福祉車帶技能 `{1}`;需福祉的單帶技能 `{1}` → 只能上福祉車;非福祉單無此技能 → 任意車
  (福祉車為能力超集,閒時可兼接一般單)。

## 原則 5:司機出勤緩衝(發車提早、收車跨過)
- **政策**:服務時段 06:00–18:00(乘客上車限此區間不變),但**司機/車輛可用時段 = 05:30–18:30**
  ——可提早 30 分發車去接 06:00 早單;「18:30 前已上車」的在途趟次**可跨過 18:30 完成**(保留完成緩衝)。
- 落地:設定 `driver_margin_min`(預設 30,可於參數頁調整)。
  車輛時間窗 = `[服務起 − margin, 服務迄 + 完成緩衝]`;乘客上車窗仍限服務時段(`out_of_service` 不變)。
- 程式:`dispatcher` `veh_day_start = day_start - margin`;`veh_day_end = day_end + completion_buffer`。
- 實測(2026-06-22):05:30 發車讓 06:00 桃園早單排入,未派 5→4、已派 202→203(用 38 車)。

## 原則 6:只派給「當天有司機」的車
- 班表(roster)是以**車輛**排的,可能有「有班表但沒司機」的車;沒司機的車實際開不了。
- 派遣只納入**當天有司機**的車(司機預設車 `Driver.vehicle_id` + 當日輪車 `DriverVehicleAssignment`)。
- 安全退路:若整個車隊查無任何司機對應,則不過濾(避免整日無法派遣)。
- 程式:`dispatcher._vehicles_with_driver` 過濾 `duty`。
- 實測 2026-06-23:過濾前 35 車(6 台無司機)→ 過濾後 **32 車全有司機、仍 240 單全派**(更省車)。
- **無班表退路**:當完全無值勤班表(未建 roster)時,改以「全部 active 且有司機之車」為可用池
  (而非保守視為全不出勤),避免空名冊導致大量未派;有班表時仍只用當日出勤車。
  實測 6/24:退路前可用 16 車、未派 35 → 退路後 35 車、未派 2。

## 原則 7:最長乘車時間上限(防止共乘把人載太久)
- **問題**:原模型只限上車時間窗、不限下車;VROOM 為省車會把長程下車拖到收車前
  (實例:乘客 09:57 上車、17:36 下車,在車 7.6 小時)。數學合法、現實不可接受。
- **政策**:每筆下車時間窗上限 = 上車窗末 + `max(40 分, 直達車程 × 倍率 + 緩衝)`;
  乘車超過此上限的解視為不可行,VROOM 會改派/留未派而非久載。
- 落地:設定 `max_ride_factor`(預設 1.8)、`max_ride_grace_min`(預設 30);
  程式 `dispatcher._max_ride_upper` / `comparison._max_ride_upper`,**即時與對比皆套用**。
- 取捨:固定車數下未派略增(長程往返需專車),但杜絕不合理久載;即時派遣有整個名冊可吸收。

## 原則 8:每趟作業時間用歷史校準(車行 × 福祉),不再全域固定
- **問題**:原全域固定「每趟 40 分(前置 20+後置 20)」對短程密集接駁過重——金山在地接駁
  一台車一天 19 趟,40 分/趟 = 12.7h 爆 8h 上限,逼出多車;人工真實上下車僅 ~20 分。
- **政策**:每趟作業時間 = 該**車行 × 是否福祉單**的歷史校準值(背靠背連續趟反推,
  扣車程與空車);**樣本不足(<30 趟對)退回全域**。距離由 OSRM 車程動態處理,**不綁距離**;
  區域(山區/都會)差異由「分車行各自校準」吸收,**不硬編係數**。
- 落地:`calibration.py` + `fleet_calibration` 表(migration 0026);`/dispatch/calibration` 端點 +
  參數設定頁「🧪 每趟作業時間—歷史校準」卡;`dispatcher` 與 `comparison` 皆依車行×福祉取每趟作業秒數。
- 實證(全域):一般 19.4 分、福祉 21.5 分(N=10,705 趟對);校準後神同行 3 台車 0 未派,
  完全重現人工。詳見 `findings-calibration-2026-06-23.md`。
- **重要**:此校準把對比從「偏保守的下限」拉回真實——全量省車率 **↓18.7% → ↓35.9%**。

## 原則 9:固定趟為「既定區塊」— 各釘各車、計整趟佔用時間
- **問題**:固定趟(校車/日照/洗腎等已談妥行程)是一車多人的委派,班表常只記頭尾、時間為假設;
  若當一般散單最佳化,VROOM 會把「同到某校」的多條固定趟亂併成少數車(虛報省車、忽略實況)。
- **政策**:訂單 `occupancy_min` 有值即視為**既定區塊**——以唯一技能**釘到指定車**、以整趟佔用時間為
  服務時長、不受最長乘車上限、**各固定趟各釘各車不互併**;正常單在其空檔最佳化。
- 落地:`orders.occupancy_min`(migration 0027);`comparison.compare_day` 釘車邏輯;
  固定行程逐條參數於 `fixed_route`(migration 0028)維護,缺項回退參數設定預設
  (`fixed_route_blocks.resolve_params`,佔用時間以 OSRM 頭尾 + 多站緩衝估算)。
- 驗證(6/23 校車):當散單亂併 17→10(失真)/ 既定區塊 17→16(正確)。詳見 `findings-fixed-route-2026-06-23.md`。

## 旋鈕:省車率鬆緊(`service_time_factor` + 上車時間窗)
- 研究證實**省車率幾乎全由「時間間距」決定**:每趟作業時間 ×`service_time_factor`(校準×係數,**不改校準真值**)
  與上車時間窗 `pickup_window_min`;工時上限/最長乘車/完成緩衝在真實需求下幾乎無作用。
- 可於參數設定即時調(8%–60% 對照轉盤見研究);系統預設目前定在約 **14% 省車**(窗 9 分、係數 1.55,保守貼近人工)。

## 不變的硬約束(護欄)
時間窗(上車 `[T, T+window]`、服務時段 06–18)、**最長乘車時間上限**、座位容量、8h 工時上限、
進行中行程鎖原車、固定行程釘指定車 —— 皆為**硬約束**,不因軟性原則放寬。

> 註:原則 7(乘車上限)、原則 8(歷史校準工時)**對比引擎 `comparison.py` 亦同步套用**
> (與舊版「對比沿用固定方法學」不同)——因為這兩項是「把不真實的解/參數修正成真實」,
> 套用後對比才反映真實營運。其餘軟性原則(1/2/3/5/6)仍僅規範即時派遣。
