# 派遣系統評估報告 — 2026/06/24(固定趟既定區塊全做後)

**對象**:DrFish 長照接送派遣系統 ｜ **範圍**:本輪客戶回饋改善 + 即時派遣/歷史對比評估
**資料**:雙北/基隆/神同行真實歷史 47,432 單(2024-11~2026-06)+ 6/24 當日 248 單 ｜ 路網:自架 OSRM

---

## 一、摘要

本輪依客戶對「固定趟(校車/特教/日照/洗腎)」的營運回饋,把固定趟建模為**既定區塊(committed block)**:
以真實佔用時間佔住司機整個時段、各路線各釘各車、不互相併車,但**仍可用空檔接非固定散單**。

| 指標 | 結果 |
|---|---|
| 6/24 即時派遣 | **248 / 248 全派**(用車 39) |
| 固定趟既定區塊 | 40 筆,校車/特教各司機**一時段一趟** |
| 客戶 3 點回饋 | **全數達成**(見第四節) |
| 歷史全量省車率(現行保守預設) | **14.7%**(窗9/係數1.55,777 車行-日) |
| 省車率可調範圍 | 8%–60% 旋鈕(校準真值 factor=1.0 為 ~34.5%) |

---

## 二、背景:本輪客戶回饋

1. **固定趟是整時段共乘委派**:校車/特教一車多人,班表常只記頭尾、時間為假設;一個時段只能執行**一條**固定趟,不可兩條,否則無法準時到校/返家。
2. **可接非固定散單**:司機接了一條(整時段)固定趟後,**時間允許仍可接其他非固定散單**填空檔。
3. **到校/時窗**:台北約 08:30 到校、新北約 09:00;特教到校 08:00–09:00、返家 15:30–16:30。
4. **同駕駛**:固定趟盡量綁同一駕駛,除非有狀況才換。

### 改善前的問題(實證)
6/24 初次派遣時,固定趟以 ~20 分普通單處理 + 任意插單:校車司機早上被排 **3–5 趟**(含一條 180 分校車),**無法準時到校**;且成對路線(成德-1/-2 等)因 keyword 過廣被併到**同一台車**。

---

## 三、本輪改善與技術實現

| 改善 | 實現 |
|---|---|
| **既定區塊 + 真實佔用時間** | 固定趟 `orders.occupancy_min` 帶入來源「服務分鐘」(校車多為 180 分);`dispatcher`/`compare_day` 以其為整趟服務時長、不設最長乘車上限 → 佔住整時段、期間不插單 |
| **成對路線各釘各車** | 固定行程 `keyword` 由廣義地點(「成德國中」)改為**完整路線名**(成德國中-1 / -2…),避免 -1/-2 互相命中被併車;時段(早/午後)分流仍有效 |
| **可接非固定散單** | 區塊只佔「該趟所需時間」,VROOM 在剩餘空檔自然插入非固定散單(時間可行才插) |
| **無班表退路** | 未建值勤名冊時改用「全 active 且有司機之車」為可用池(避免空名冊大量未派) |
| **省車率主旋鈕** | `service_time_factor`(校準×係數)+ 上車時間窗;省車鬆緊 8%–60% 可調,預設保守 14% |

> 設計原則:固定趟「一時段一趟」由**真實佔用時間 + VROOM 時間可行性**自然達成,**不硬性禁止**;短程固定趟(洗腎/工廠)時間放得下仍可多接,避免誤殺。

## 四、結果評估

### 4.1 6/24 即時派遣
- **248 單全派(100%)**,用車 39(福祉單 92、一般單 156);固定趟既定區塊 40 筆。
- **校車/特教(occ 120–180)司機:一時段一趟**(如梁銘漢成德-1、王俊凱潭美-1…早 1 趟、午後 1 趟),其餘空檔接非固定散單(20 個時段有 gap-fill)。
- 成對路線各釘各車並時段分流:成德國中-2 早(吳奇龍/RDP-8365)、午後(林泰吉/REA-7983)。

### 4.2 客戶回饋逐項達成

| # | 回饋 | 狀態 |
|---|---|---|
| 1 | 一時段只能一條(整時段)固定趟 | ✅ 真實佔用時間 + 各釘各車 |
| 2 | 仍可接非固定散單填空檔 | ✅ 空檔 gap-fill(不硬禁) |
| 3 | 到校/返家時窗(台北08:30/新北09:00、特教08-09/15:30-16:30) | ◷ 以來源服務分鐘+起始時段近似;可再逐條/區域精修 |
| 4 | 固定趟綁同駕駛 | ✅ 固定行程 driver/車牌釘定,當天可換 |

### 4.3 歷史全量對比(省車率)
- 現行**保守預設(窗9/係數1.55)**:777 車行-日,人工 7,699 → 系統 6,565 = **省車 14.7%**。
- 省車率為**可調旋鈕**(時間間距:作業係數 + 上車窗),研究證實工時上限/最長乘車/緩衝在真實需求下幾乎無作用。
- 校準真值(factor=1.0)時 ~34.5%;8%–60% 對照轉盤見 `dispatch-principles.md`。

## 五、效益量化

- **準確性**:固定趟不再被當散單亂併(校車省車假象 17→10 已修正為合理 17→16);即時派遣達真實可執行的 248/248。
- **彈性**:單一旋鈕在 8%–60% 間調整省車積極度,**不竄改歷史校準真值**;預設保守 14% 貼近人工、利於試跑導入。
- **真實資料驗證**:全程以 4 車隊 ~4.7 萬筆真實歷史 + 連續三日(6/22–24)班表/訂單驗證,非合成資料。

## 六、限制與後續建議

1. **到校時窗精修**(回饋3):目前以「服務分鐘 + 起始時段」近似;若要嚴格保證台北08:30/新北09:00 到校,可於固定行程逐條維護「到校時窗」或以區域(車行)預設下車時間窗。
2. **固定趟辨識**:目前以乘客姓名/路線名比對(假單名=路線名);未來真實乘客單需「地點/case 比對」才能自動歸入固定趟(現為保存規則,見 open-decisions)。
3. **資料一致性護欄**:建議於固定行程維護頁示警「福祉單 vs 指定車車型不符」「同一司機同時段多條固定趟」,提早讓行控修正。
4. **值勤班表**:長期應建 roster(現走無班表退路);建好後可更精準控管當日出勤與工時。
5. **個資合規**:每日派車表含乘客/地址,僅放 `reports/`(gitignore),正式上線前落實多租戶隔離與個資政策。
